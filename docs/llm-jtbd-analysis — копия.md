# LLM Jobs to Be Done — PropTech News Monitor

**Документ:** Анализ задач, которые могут и должны выполняться с помощью LLM
**Проект:** PropTech News Monitoring Agent (Avito Real Estate)
**Дата:** 2026-04-06
**Методология:** Анализ кода, конфигурации, спецификации агента и промптов

---

## Контекст и подход

Проект представляет собой агента для мониторинга рынка недвижимости, который проходит следующие стадии пайплайна:

```
Сбор источников → Извлечение контента → Дедупликация → Классификация →
Скоринг → Суммаризация → Сборка дайджеста → Доставка → Обратная связь
```

На каждой из этих стадий есть задачи двух типов:
- **Детерминированные** — их лучше решать кодом (HTTP-запросы, парсинг RSS, отправка в Telegram)
- **Интеллектуальные** — именно здесь LLM создаёт принципиальную ценность

Ниже — полный каталог LLM-задач (JTBD), выделенных из кода проекта.

## JTBD по стадиям пайплайна

---

### Стадия 1. Сбор и обнаружение источников

---

#### JTBD-01 — Генерация поисковых запросов ⚠️

**Суть задачи:** Агент использует WebSearch для поиска новостей в реальном времени. В конфиге прописаны 6 статичных запросов. LLM должен уметь генерировать адаптивные запросы, которые учитывают актуальный контекст: что уже было в прошлых дайджестах, какие темы стали «горячими», какой период охватывает текущий запуск.

**Почему нельзя решить кодом:** Статичные запросы быстро устаревают. LLM понимает семантику предметной области и может формулировать запросы, которые вытащат нужный контент — даже если ключевые слова изменились.

**Вход:** История последних дайджестов, текущая дата, список фокусных тем из config
**Выход:** 5–10 поисковых запросов с пояснением логики каждого
**Текущий статус:** Запросы захардкожены в `monitoring.yaml`, не адаптируются

---

#### JTBD-02 — Оценка качества и актуальности источника ❌

**Суть задачи:** Источники меняются: RSS-ленты ломаются, сайты меняют структуру, редакционный фокус смещается. LLM должен периодически оценивать, даёт ли конкретный источник нужный сигнал, и рекомендовать: продолжать мониторить, снизить частоту, заменить на другой.

**Почему нельзя решить кодом:** Решение требует понимания того, о чём пишет источник и насколько это соответствует бизнес-фокусу команды. Код может лишь проверить доступность URL.

**Вход:** Последние 20 статей источника, его заявленный фокус, текущий приоритет
**Выход:** Оценка (keep / downgrade / replace) + короткое обоснование
**Текущий статус:** В `rss-api-audit.md` есть ручной аудит, но автоматической оценки нет

---

#### JTBD-03 — Предложение новых источников ❌

**Суть задачи:** Рынок proptech постоянно меняется: появляются новые аналитики, исследовательские центры, блоги компаний. LLM должен по упомянутым в дайджестах ссылкам и источникам предлагать кандидатов на добавление в `monitor-list.json`.

**Почему нельзя решить кодом:** Требует понимания, чем источник полезен для конкретной аудитории (Avito Real Estate), а не просто того, что он существует.

**Вход:** Список URL, упомянутых в последних N дайджестах, но не входящих в текущий мониторинг
**Выход:** Список кандидатов с приоритетом, типом контента и обоснованием добавления

---

### Стадия 2. Извлечение и очистка контента

---

#### JTBD-04 — Извлечение структурированного контента из «грязного» HTML ⚠️

**Суть задачи:** 9 из 19 источников требуют Chrome-скрапинга (SPA, нет RSS). После получения HTML-страницы нужно извлечь именно статьи/новости, а не меню, рекламу, футер и boilerplate. LLM справляется с этим значительно лучше CSS-селекторов, которые ломаются при любом редизайне.

**Почему нельзя решить кодом:** Правила для CSS-селекторов нужно поддерживать вручную для каждого сайта. LLM понимает семантику контента.

**Вход:** Сырой HTML или text-контент страницы
**Выход:** Список `{title, url, date, summary}` — только новостные материалы
**Текущий статус:** В `rss-api-audit.md` описаны regex-подходы для конкретных сайтов — хрупко

---

#### JTBD-05 — Нормализация дат и метаданных ⚠️

**Суть задачи:** Разные источники используют разные форматы дат, локальные временные зоны, неполные метаданные. LLM надёжнее парсит «3 days ago», «Apr 4, 2026», «вчера», «Q1 2026» и приводит всё к единому ISO-формату.

**Почему нельзя решить кодом:** Форматы слишком разнообразны и не всегда машиночитаемы. Требуется понимание контекста.

**Вход:** Текстовое поле даты из источника + текущая дата
**Выход:** Нормализованная дата ISO 8601

---

### Стадия 3. Дедупликация

---

#### JTBD-06 — Семантическая дедупликация ⚠️

**Суть задачи:** Текущий механизм дедупликации в `dedupe.json` работает по canonical URL и порогу сходства заголовков (0.9). Этого недостаточно: одна и та же новость (например, «Zillow запустил conversational search») может появиться на 5 разных сайтах с разными заголовками и URL. LLM должен определять, что это один и тот же факт.

**Почему нельзя решить кодом:** String similarity не понимает смысл. Нужно семантическое сравнение.

**Вход:** Новая статья + корпус уже виденных статей за lookback-период
**Выход:** `is_duplicate: true/false`, если дубль — ссылка на «главную» версию
**Текущий статус:** Только URL-matching + title similarity, семантики нет

---

### Стадия 4. Классификация и скоринг

---

#### JTBD-07 — Классификация типа сигнала ✅

**Суть задачи:** Каждой новости присваивается тип из фиксированной таксономии: `product_launch`, `ai_feature`, `monetization_change`, `m_and_a_or_investment`, `seller_or_agent_tooling`, `traffic_or_market_share_shift` и ещё 4 типа. Это критично для последующей фильтрации и маршрутизации.

**Почему нельзя решить кодом:** Ключевые слова не работают. «Zillow Preview» — это `product_launch`, а не просто новость о Zillow. Требуется понимание смысла события.

**Вход:** Заголовок + лид-абзац статьи
**Выход:** Тип сигнала из таксономии + confidence
**Текущий статус:** Описано в `monitor-list.json` и промпте, но без явного step-by-step

---

#### JTBD-08 — Многомерный скоринг релевантности ✅

**Суть задачи:** Ключевой интеллектуальный джоб. Каждая новость оценивается по 5 измерениям: `marketplace_relevance` (35%), `event_scale` (25%), `portability_to_avito` (20%), `urgency` (10%), `novelty` (10%). Итоговый score определяет, попадёт ли новость в дайджест.

**Почему нельзя решить кодом:** Это суждение, требующее понимания бизнес-контекста Avito, природы события и возможных цепочек последствий.

**Вход:** Полный текст статьи + система весов из config + описание аудитории
**Выход:** Числовой score (0–100) + оценки по каждому измерению отдельно + confidence
**Текущий статус:** Описано в `agent-spec.md` и `news_analyst.md`, веса в конфиге ✅

---

#### JTBD-09 — Оценка уровня срочности (breaking alert) ✅

**Суть задачи:** Агент каждый час проверяет, нет ли события с score ≥ 85, требующего немедленной отправки alert. LLM должен уметь отделять «большую» новость от обычной независимо от времени публикации.

**Почему нельзя решить кодом:** Срочность — это смысловое суждение. «Rightmove подала в суд на Zoopla» — это breaking alert. «Rightmove опубликовала квартальный отчёт» — нет.

**Вход:** Статья, её score по основной шкале
**Выход:** `is_breaking: true/false` + short rationale
**Текущий статус:** Критерий score ≥ 85 в конфиге есть, логика LLM-оценки описана частично

---

#### JTBD-10 — Обнаружение слабых сигналов ✅

**Суть задачи:** Статьи с score 40–55 не попадают в основной дайджест, но могут быть ранними индикаторами тренда. LLM должен уметь их распознать и включить в раздел «Слабые сигналы» с соответствующим контекстом.

**Почему нельзя решить кодом:** «Слабость» сигнала — это не просто низкий score, а суждение о том, что тема ещё не стала мейнстримом, но уже заслуживает внимания.

**Вход:** Статьи с score 40–55
**Выход:** Список слабых сигналов с кратким объяснением, почему они важны сейчас
**Текущий статус:** Логика описана в `agent-spec.md` и видна в примере дайджеста ✅

---

### Стадия 5. Суммаризация и синтез

---

#### JTBD-11 — Суммаризация новости ✅

**Суть задачи:** Каждая новость в дайджесте представлена в виде 2–3 bullet-points: что случилось, почему важно, что это значит для Avito. Это базовая, но критически важная LLM-задача.

**Почему нельзя решить кодом:** Требует понимания контекста, умения выделить главное и сформулировать кратко и точно.

**Вход:** Полный текст статьи + профиль аудитории (Avito Real Estate PM/Strategy)
**Выход:** Структурированное резюме: `{what_happened, why_it_matters, avito_implication}`
**Текущий статус:** Описано в `news_analyst.md` ✅

---

#### JTBD-12 — Анализ импликаций для Avito ✅

**Суть задачи:** Самый ценный джоб из всех. Для каждой новости LLM должен ответить на вопрос: «Что это конкретно означает для Avito Real Estate?». Не общие слова, а конкретные гипотезы: «Тест conversational search entry point», «Риск при повышении тарифов».

**Почему нельзя решить кодом:** Требует глубокого понимания бизнес-модели Avito, конкурентного ландшафта и цепочек причинно-следственных связей.

**Вход:** Суммаризация новости + описание бизнеса Avito из системного промпта
**Выход:** 1–2 конкретные гипотезы/рекомендации с маркером `[гипотеза]` при неопределённости
**Текущий статус:** Описано в промпте `news_analyst.md` ✅, но инструкция «избегать гиперболизации» требует доработки

---

#### JTBD-13 — Синтез трендов из нескольких новостей ❌

**Суть задачи:** Если за неделю вышли 4 статьи про AI-поиск в недвижимости (Zillow, Realtor.com, Domain, Hemnet) — это тренд, заслуживающий отдельного вывода. LLM должен агрегировать сигналы и формулировать надстраивающий инсайт уровнем выше отдельных статей.

**Почему нельзя решить кодом:** Паттерн-матчинг через теги не улавливает суть — нужно семантическое рассуждение о том, что объединяет события.

**Вход:** Все статьи недельного дайджеста + история предыдущих 4 недель
**Выход:** 1–3 синтезированных тренда с поддерживающими фактами
**Текущий статус:** Отсутствует. Видно в примере дайджеста только косвенно в секции «What to Watch»

---

#### JTBD-14 — Генерация action items для команды ✅

**Суть задачи:** В конце каждого дайджеста — раздел «Что смотреть» с конкретными рекомендациями для команды (PM, Strategy, Commercial). LLM должен не просто пересказать новости, а сформулировать следующие шаги.

**Почему нельзя решить кодом:** Требует понимания того, что является actionable для конкретной команды в конкретном контексте.

**Вход:** Топ-N новостей + роль аудитории (Product / Strategy / Commercial)
**Выход:** 3–5 конкретных action items с привязкой к источнику
**Текущий статус:** Описано и видно в примере дайджеста ✅

---

#### JTBD-15 — Контекстуализация (память о прошлых событиях) ❌

**Суть задачи:** Новость «CoStar расширяет иск против Zillow» имеет смысл только если читатель знает, что изначальный иск был подан в Q3 2025. LLM должен добавлять исторический контекст из архива дайджестов.

**Почему нельзя решить кодом:** Нужно семантическое сопоставление новой новости с историей, а не просто поиск по keyword.

**Вход:** Новая статья + архив дайджестов (`.state/archive/`)
**Выход:** Краткий контекст «продолжение темы X от [дата]» + ссылка на предыдущий дайджест
**Текущий статус:** Архив ведётся в `delivery-log.json`, но никак не используется как контекст для LLM

---

### Стадия 6. Сборка дайджеста

---

#### JTBD-16 — Ранжирование и финальный отбор историй ✅

**Суть задачи:** Из всех прошедших порог статей нужно выбрать 5–7 для основного дайджеста и 4 для слабых сигналов. Это не просто «взять топ-7 по score» — нужно учесть разнообразие тем, баланс компаний, региональный охват.

**Почему нельзя решить кодом:** Сортировка по числу не даёт редакционного качества. «7 новостей про Zillow» хуже, чем «5 разных компаний».

**Вход:** Скорированный пул статей + правила из config (max top items, weak signals quota)
**Выход:** Финальный отранжированный список с обоснованием выбора
**Текущий статус:** Описано в `agent-spec.md`, критерии в конфиге ✅

---

#### JTBD-17 — Написание вступительного резюме дайджеста ⚠️

**Суть задачи:** Дайджест должен начинаться с 2–3 предложений, которые захватывают суть недели/дня — «главная тема» или «настроение рынка». Это редакторский джоб.

**Почему нельзя решить кодом:** Требует синтетического суждения о том, что важнее всего в подборке.

**Вход:** Финальный список историй дайджеста
**Выход:** Вступительный абзац (2–4 предложения) на русском языке
**Текущий статус:** В примере дайджеста вступление есть, но в промпте не специфицировано

---

#### JTBD-18 — Адаптация дайджеста под разные аудитории ❌

**Суть задачи:** Промпт в `agent-spec.md` (MVP v3) упоминает multi-audience personalization: Strategy vs Product vs Commercial. Один и тот же набор новостей нужно подавать по-разному: Strategy хочет M&A и рыночные сдвиги, Product — product launches и UX, Commercial — monetization и agent tools.

**Почему нельзя решить кодом:** Требует понимания того, что релевантно для каждой роли и как переформулировать импликации.

**Вход:** Финальный дайджест + профиль аудитории
**Выход:** 3 варианта TL;DR (по 3–5 строк) для разных команд
**Текущий статус:** Запланировано в MVP v3, не реализовано

---

#### JTBD-19 — Форматирование и сплитование для Telegram ⚠️

**Суть задачи:** Telegram имеет лимит 4096 символов. Агент сплитует сообщения (порог в конфиге: 3800 символов для дайджеста, 2500 для alerts). LLM должен разбивать текст семантически — не посередине истории, а между историями.

**Почему нельзя решить кодом:** Механический split по символу разрывает контекст. LLM понимает границы смысловых блоков.

**Вход:** Собранный дайджест + лимит символов
**Выход:** Список сообщений, разбитых по смысловым блокам + правильный Markdown для Telegram
**Текущий статус:** Конфиг содержит лимиты, но логика split не описана

---

### Стадия 7. Обратная связь и обучение

---

#### JTBD-20 — Интерпретация обратной связи пользователя ❌

**Суть задачи:** MVP v3 предполагает feedback loop (thumbs up/down). Но даже без голосований команда даёт неявную обратную связь: кто-то пересылает новость, кто-то отвечает на сообщение в Telegram. LLM должен интерпретировать эти сигналы и корректировать веса релевантности.

**Почему нельзя решить кодом:** Сигналы обратной связи неформальные, разнородные — нужен смысловой анализ.

**Вход:** Telegram-активность на дайджест (реакции, репосты, ответы), история дайджестов
**Выход:** Рекомендации по корректировке scoring weights или фокусных тем
**Текущий статус:** Запланировано в MVP v3, не реализовано

---

#### JTBD-21 — Оценка качества предыдущих дайджестов ❌

**Суть задачи:** Периодически (например, ежемесячно) LLM должен анализировать архив дайджестов и отвечать на вопрос: «Насколько прогнозы и импликации подтвердились?». Это позволяет улучшать промпт и scoring weights.

**Почему нельзя решить кодом:** Требует семантического сравнения «что мы предсказали» с «что произошло».

**Вход:** Архив дайджестов за месяц + свежие новости
**Выход:** Ретроспективный отчёт: что подтвердилось, что нет, почему
**Текущий статус:** Отсутствует

# Дизайн бенчмаркинга LLM — PropTech News Monitor

**Документ:** Методология подготовки кейсов для оценки качества LLM по 16 JTBD
**Проект:** PropTech News Monitoring Agent (Avito Real Estate)
**Дата:** 2026-04-06
**Читателям:** нужен контекст из `docs/llm-jtbd-analysis.md`

---

## 1. Зачем и что мы измеряем

Цель бенчмаркинга — ответить на конкретные вопросы принятия решений:

- Насколько хорошо текущий промпт справляется с задачей? (baseline)
- Какую модель использовать для каждого JTBD? (model selection)
- Как изменение промпта влияет на качество? (regression testing)
- Где LLM системно ошибается и почему? (failure analysis)

Бенчмарк — это не разовая акция, а **живой инструмент**: набор тест-кейсов, который пополняется по мере работы агента и позволяет проверять любое изменение промпта за минуты.

---

## 2. Три составляющих каждого JTBD-бенчмарка

Для каждой из 16 задач нужно подготовить три артефакта:

### 2.1 Тест-кейсы (Input Dataset)
Набор реалистичных входных данных, покрывающих:
- **Типичные случаи** — что встречается чаще всего
- **Граничные случаи** — где задача неоднозначна
- **Негативные случаи** — где правильный ответ — «не делать» (отклонить, не добавлять)

Минимальный объём: **20 кейсов** для generation-задач, **50 кейсов** для classification-задач.

### 2.2 Golden Set
Эталонные ответы, относительно которых оценивается LLM. Создаются двумя способами:
- **Аннотация людьми** — эксперт домена размечает входные данные вручную
- **LLM-assisted + human review** — LLM генерирует черновик, эксперт исправляет

Для задач с субъективными ответами (генерация текста) golden set — это не «единственно верный ответ», а **примеры ответов с оценкой A/B/C** и объяснением разницы между ними.

### 2.3 Скоринговая методология
Как сравнивать output LLM с golden set. Три метода в зависимости от типа задачи:

| Метод | Когда применять | Примеры метрик |
|-------|-----------------|----------------|
| **Hard metrics** | Классификация, бинарные решения | Precision, Recall, F1, Accuracy |
| **LLM-as-Judge** | Генерация текста, суммаризация | Rubric score 1–5 по каждому критерию |
| **Human eval** | Сложные аналитические задачи, долгосрочные прогнозы | Ранжирование ответов (A vs B), Likert-шкала |

### JTBD-02 — Оценка качества и актуальности источника

**Группа:** D (Human eval)
**Объём dataset:** 15–20 источников

#### Входные данные
Для каждого тест-кейса:
```json
{
  "source_name": "Mike DelPrete",
  "declared_focus": "proptech industry analysis, global real estate portals",
  "priority_current": 1,
  "recent_articles": [
    {"title": "...", "date": "...", "summary": "..."},
    ...  // последние 15–20 статей
  ],
  "avito_focus_areas": ["AI search", "monetization", "marketplace model"]
}
```

Откуда брать: реальные источники из `monitor-list.json` + их свежий контент (сделать снэпшот).
Включить: 5 источников «очевидно хороших», 5 «средних», 5 «сомнительных» (нет proptech-фокуса или устарели).

#### Критерии оценки

| Критерий | Вес | Что измеряем |
|----------|-----|--------------|
| Корректность решения keep/downgrade/replace | 40% | Совпадение с экспертом |
| Качество обоснования | 35% | Конкретность, опора на факты из статей |
| Практичность рекомендации | 25% | Есть ли альтернатива при replace |

#### Golden Set
1. Взять 15–20 реальных источников
2. Каждый независимо оценивают два эксперта (например, PM + стратег Avito)
3. Разногласия разбираются и фиксируется финальное решение + обоснование
4. Результат: JSON `{source_id, decision, rationale, alternative_if_replace}`

#### Методология оценки
- **Основная метрика:** Cohen's Kappa между LLM и golden set по трёхклассовой классификации
- **Дополнительно:** LLM-as-Judge оценивает качество обоснования по рубрике (1–5)
- **Порог качества:** Kappa ≥ 0.65 (substantial agreement), rubric ≥ 3.5/5

#### Edge cases
- Источник технически доступен, но изменил редакционный фокус → правильный ответ: downgrade/replace
- Источник временно недоступен, но исторически ценный → не replace только на основе недоступности

### JTBD-03 — Предложение новых источников

**Группа:** D (Human eval)
**Объём dataset:** 10 сценариев

#### Входные данные
```json
{
  "context": "Сентябрь 2026, фокус на AI-поиск в аренде",
  "recent_digest_urls": [
    // 30–50 URL, которые упоминались в последних дайджестах, но не в monitor-list
  ],
  "current_monitor_list": [...],  // краткие описания текущих источников
  "avito_focus_areas": [...]
}
```

Откуда брать: вручную собрать URL из `dedupe.json` и примера дайджеста, которых нет в `monitor-list.json`. Дополнить синтетически (5 «очевидных» кандидатов + 5 нерелевантных).

#### Критерии оценки

| Критерий | Вес | Что измеряем |
|----------|-----|--------------|
| Precision предложений | 40% | Доля предложенных источников, реально достойных мониторинга |
| Recall | 30% | Доля «правильных» кандидатов, которые LLM предложил |
| Качество обоснования | 30% | Почему источник важен, что он покрывает |

#### Golden Set
1. Собрать 40–60 URL-кандидатов
2. Эксперт вручную классифицирует каждый: `add_priority_1 / add_priority_2 / skip`
3. Golden = список из ~15–20 рекомендованных источников с объяснением

#### Методология оценки
- **Precision@5, Precision@10:** топ-5 и топ-10 предложений LLM vs golden
- **LLM-as-Judge:** оценка обоснования по рубрике (специфичность, опора на контент источника)
- **Порог:** Precision@10 ≥ 0.6

### JTBD-06 — Семантическая дедупликация

**Группа:** A (Hard metrics)
**Объём dataset:** 80–100 пар статей

#### Входные данные
```json
{
  "article_new": {
    "title": "Zillow Debuts Conversational Home Search",
    "source": "TechCrunch",
    "date": "2026-04-03",
    "body": "..."
  },
  "corpus": [
    {
      "title": "Zillow's AI Search Goes Live",
      "source": "Inman",
      "date": "2026-04-02",
      "body": "..."
    },
    ...  // 5–15 статей из lookback-окна
  ]
}
```

Типы кейсов (распределение):
- 30% — явные дубли (одно событие, разные источники, схожие заголовки)
- 30% — семантические дубли (одно событие, разные заголовки и формулировки)
- 25% — похожая тема, но разные события (не дубль)
- 15% — случайные совпадения слов (ложные позитивы)

Откуда брать: `dedupe.json` содержит 23 реальных записи → расширить до 80–100 через:
а) реальный запуск агента за 2–3 недели
б) ручная генерация синтетических пар по существующим статьям

#### Критерии оценки
- **Precision** (не помечать уникальные новости как дубли)
- **Recall** (находить все семантические дубли)
- **F1-score** (баланс)
- Отдельно: F1 на подгруппе «семантические дубли» (самый сложный случай)

#### Golden Set
1. Собрать пары статей из реальных запусков агента
2. Разметить вручную: `{pair_id, is_duplicate: true/false, duplicate_type: exact/semantic/false_positive, rationale}`
3. Два независимых аннотатора, разногласия разбираются

#### Методология оценки
- Метрики: Precision, Recall, F1 на всём датасете + отдельно по типам
- **Порог:** F1 ≥ 0.80 на «семантических дублях»
- Анализировать матрицу ошибок: какой тип дублей пропускается чаще

#### Edge cases
- Одно событие (IPO Zillow) освещается 5 дней подряд → каждый день новые детали, не дубль
- Статья-обзор, которая ссылается на событие недельной давности → не дубль

### JTBD-07 — Классификация типа сигнала

**Группа:** A (Hard metrics)
**Объём dataset:** 100 статей

#### Входные данные
```json
{
  "title": "Realtor.com Overhauls Search With AI-Powered Filters",
  "lead": "Move Inc. today announced...",
  "source": "Inman"
}
```

Таксономия (9 классов из `monitor-list.json`):
`product_launch`, `ai_feature`, `monetization_change`, `inventory_or_supply_move`,
`seller_or_agent_tooling`, `partnership`, `m_and_a_or_investment`,
`org_or_leadership_change`, `traffic_or_market_share_shift`

Распределение: минимум 8–10 примеров на каждый класс + 10–15 многозначных случаев (статья подходит под 2 класса).

#### Критерии оценки
- **Accuracy** (топ-1 предсказание)
- **Top-2 Accuracy** (правильный ответ в топ-2 предсказаний)
- **Per-class F1** — особенно важно для редких классов (`org_or_leadership_change`)
- Отдельно: accuracy на многозначных случаях

#### Golden Set
1. Взять реальные статьи из `dedupe.json` + пример дайджеста (у них уже есть типы)
2. Дополнить синтетическими примерами по 2–3 на недопредставленные классы
3. Разметка: эксперт ставит primary + optional secondary класс
4. Формат: `{article_id, primary_type, secondary_type_optional, confidence, rationale}`

#### Методология оценки
- Macro-F1 (равный вес всех классов)
- Confusion matrix для анализа систематических ошибок
- **Порог:** Accuracy ≥ 0.82, Macro-F1 ≥ 0.75

#### Edge cases
- `ai_feature` vs `product_launch` — AI-функция внутри большого продуктового запуска
- `partnership` vs `m_and_a_or_investment` — стратегическое партнёрство с опционом на поглощение

### JTBD-08 — Многомерный скоринг релевантности

**Группа:** B (LLM judge + correlation)
**Объём dataset:** 40–50 статей

#### Входные данные
```json
{
  "article": {
    "title": "...",
    "body": "...",
    "source": "...",
    "date": "..."
  },
  "scoring_dimensions": {
    "marketplace_relevance": {"weight": 0.35, "description": "..."},
    "event_scale": {"weight": 0.25, "description": "..."},
    "portability_to_avito": {"weight": 0.20, "description": "..."},
    "urgency": {"weight": 0.10, "description": "..."},
    "novelty": {"weight": 0.10, "description": "..."}
  },
  "avito_context": "..."  // описание бизнеса из agent-spec
}
```

Распределение по score-диапазонам:
- 10 статей с ожидаемым score 80–100 (явно высокий приоритет)
- 15 статей с score 55–79 (средний приоритет, попадут в дайджест)
- 10 статей с score 40–54 (слабые сигналы)
- 10 статей с score < 40 (нерелевантны)
- 5 статей «пограничные» (score около 55 — порога включения в дайджест)

#### Критерии оценки

| Критерий | Вес | Что измеряем |
|----------|-----|--------------|
| Ранговая корреляция с экспертом (Spearman ρ) | 40% | Насколько LLM расставляет статьи в том же порядке |
| Accuracy правильного «бакета» | 35% | Попадает ли в нужный диапазон (высокий / средний / слабый / нет) |
| Обоснование по каждому измерению | 25% | LLM-as-Judge: конкретность и корректность объяснения |

#### Golden Set
1. Отобрать 40–50 статей из реальных запусков агента + синтетика
2. Два эксперта независимо ставят score по каждому измерению (0–10) + итоговый взвешенный score
3. Разногласия > 15 баллов разбираются и фиксируется консенсус
4. Формат: `{article_id, expert_scores: {dim1, dim2, ...}, total_score, rationale}`

#### Методология оценки
- Spearman ρ между LLM-scores и expert-scores
- Confusion matrix по 4 «бакетам»
- LLM-as-Judge (отдельный промпт): оцени обоснование по шкале 1–5 за специфичность
- **Порог:** Spearman ρ ≥ 0.75, bucket accuracy ≥ 0.80

#### Edge cases
- Статья про иностранный рынок без прямой аналогии к России — portability_to_avito должен быть низким
- Старая статья (3 недели назад), но впервые попавшая в агент — urgency = 0, novelty ≠ 0

### JTBD-09 — Оценка уровня срочности (breaking alert)

**Группа:** A (Hard metrics)
**Объём dataset:** 60 статей

#### Входные данные
```json
{
  "article": {"title": "...", "body": "...", "date": "..."},
  "base_score": 78  // уже вычисленный основной score
}
```

Распределение:
- 20 статей с `is_breaking: true` (score ≥ 85 в golden + явная срочность)
- 25 статей с `is_breaking: false` (score 60–84, важно но не срочно)
- 15 статей «ложные» — высокий score, но не breaking (плановые отчёты, ожидаемые события)

#### Критерии оценки
- **Precision** (не спамить alert-ами на неважные события — ключевое!)
- **Recall** (не пропускать реальные breaking events)
- **F1-score**
- Акцент на Precision — ложный alert хуже, чем пропуск

#### Golden Set
1. Собрать реальные события из `dedupe.json` + исторические proptech-события (CoStar/Zillow иск, Rightmove lawsuit и т.п.)
2. Эксперт размечает: `{is_breaking, rationale, breaking_type: m&a/lawsuit/major_product/regulation}`
3. Включить намеренно «ловушки»: квартальный отчёт с хорошими цифрами — НЕ breaking

#### Методология оценки
- Precision, Recall, F1
- Матрица ошибок с анализом false positives (что LLM ошибочно считает breaking)
- **Порог:** Precision ≥ 0.85, Recall ≥ 0.70

### JTBD-11 — Суммаризация новости

**Группа:** C (LLM-as-Judge)
**Объём dataset:** 25–30 статей

#### Входные данные
```json
{
  "article_full_text": "...",
  "audience": "Avito Real Estate, Product/Strategy team",
  "expected_format": {
    "what_happened": "1–2 предложения",
    "why_it_matters": "1–2 предложения",
    "avito_implication": "1 конкретная гипотеза"
  }
}
```

#### Критерии оценки (рубрика)

| Критерий | Вес | Шкала 1–5 |
|----------|-----|-----------|
| Фактическая точность | 30% | 5 = нет искажений; 1 = ключевые факты перевраны |
| Полнота (всё важное отражено) | 25% | 5 = ни одного упущенного ключевого факта |
| Специфичность Avito | 25% | 5 = конкретная гипотеза; 1 = банальность типа «следить за трендом» |
| Сжатость | 20% | 5 = минимум слов, максимум смысла; 1 = вода и повторы |

#### Golden Set
1. Взять 10 реальных статей из примера дайджеста (у них уже есть суммаризации)
2. PM/стратег Avito пишет «образцовые» суммаризации для 10–15 новых статей
3. Для каждой статьи создать 2–3 варианта разного качества (A: отличный, B: приемлемый, C: плохой) с объяснением разницы → это помогает калибровать LLM-Judge

#### Методология оценки
- LLM-as-Judge (отдельный промпт с рубрикой) оценивает каждый output по 4 критериям
- Calibration check: судья должен выставить A > B > C на golden примерах
- Weighted score = сумма оценок с весами
- **Порог:** Weighted score ≥ 3.8/5.0

#### Edge cases
- Статья — перевод/пересказ более ранней статьи того же события → LLM не должен «домысливать» оригинал
- Очень короткая статья (< 100 слов) — нельзя генерировать контент, которого нет

#### Методология оценки
- LLM-as-Judge (с явной инструкцией «не принимай банальности»)
- Human spot-check 5 случайных примеров на каждый прогон бенчмарка
- **Порог:** LLM-Judge ≥ 3.7/5.0, Human spot-check: ≥ 80% примеров «приемлемо или лучше»

### JTBD-13 — Синтез трендов из нескольких новостей

**Группа:** C (LLM-as-Judge + Human eval)
**Объём dataset:** 10 наборов статей (каждый набор = 1 неделя)

#### Входные данные
```json
{
  "week_articles": [
    // 5–9 статей недельного дайджеста
  ],
  "previous_4_weeks_digests": [
    // краткие резюме предыдущих дайджестов
  ],
  "expected_output": "1–3 синтезированных тренда с supporting evidence"
}
```

Типы наборов (по 3–4 каждого):
- Набор с очевидным трендом (4+ статьи об одном феномене)
- Набор с latent трендом (связь неочевидна, но есть)
- Набор без тренда (статьи разрозненны — правильный ответ: «трендов нет»)

#### Критерии оценки (рубрика)

| Критерий | Вес | Шкала 1–5 |
|----------|-----|-----------|
| Наличие/отсутствие тренда распознано правильно | 25% | Бинарно + обоснование |
| Качество инсайта (надстраивает над отдельными статьями) | 35% | 5 = новый вывод, не очевидный из любой отдельной статьи |

#### Golden Set
1. Взять 10 реальных или реконструированных наборов статей
2. Стратег Avito пишет «правильные» тренды для каждого набора
3. Включить 3 «ловушки» — наборы без тренда (LLM должен ответить «нет общего паттерна»)

- LLM-as-Judge с рубрикой
- Бинарная точность на «есть тренд / нет тренда»

- **Порог:** Trend detection accuracy ≥ 0.80, rubric ≥ 3.5/5

```json

  "golden_metadata": {
    "annotator": "dmitry.a",
```

### Формат результата прогона
```json
{
  "run_id": "2026-04-06-claude-sonnet-4-6",
  "jtbd": "07",
  "model": "claude-sonnet-4-6",
  "prompt_version": "v1.2",
  "metrics": {
    "accuracy": 0.84,
    "macro_f1": 0.79,
    "per_class_f1": {...}
  },
  "failures": [
    {"case_id": "jtbd07-023", "predicted": "ai_feature", "expected": "product_launch", "note": "..."}
  ]
}
```

### LLM-as-Judge — ключевые принципы
1. **Судья ≠ тестируемая модель** — использовать разные модели или разные промпты
2. **Калибровка** — перед реальным прогоном судья должен правильно ранжировать A > B > C на известных примерах
3. **Structured output** — судья всегда возвращает JSON с оценкой + обоснованием
4. **Position bias** — если сравниваем два варианта, запускать судью дважды с перевёрнутым порядком
